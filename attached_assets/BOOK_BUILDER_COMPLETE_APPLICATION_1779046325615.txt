BOOK BUILDER — COMPLETE APPLICATION BLUEPRINT

================================================================================ PART 1: APPLICATION OVERVIEW
Book Builder is a full-stack AI writing and analysis platform. It connects to five proprietary LLMs (ZHI 1–5) and a suite of specialised AI services to support document rewriting, cognitive profiling, humanization, text-to-speech, graph generation, AI detection, and persistent AI chat. Every feature is accessible from a single authenticated session.

Stack: Frontend — React 18, TypeScript, Tailwind CSS, shadcn/ui, TanStack Query, Wouter Backend — Node.js, Express.js, TypeScript (ES Modules) Database — PostgreSQL via Drizzle ORM (Neon serverless) Build — Vite (frontend), esbuild (backend) Math — MathJax + KaTeX for LaTeX rendering

Authentication: Username/password. A default test user (id: 1, username: "demo", password: "password") is seeded in MemStorage on startup. All protected routes expect req.session.userId.

================================================================================ PART 5: AI MODEL ROUTING
All LLM calls in routes.ts use a switch/case pattern:

switch (model) { case 'gpt4': → openai.ts (OPENAI_API_KEY) case 'claude': → anthropic.ts (ANTHROPIC_API_KEY) case 'deepseek': → deepseek.ts (DEEPSEEK_API_KEY or XAI via base URL) case 'perplexity': → perplexity.ts (PERPLEXITY_API_KEY) case 'grok': → grok.ts (XAI_API_KEY) default: → deepseek.ts (fallback) }

Display names (getModelDisplayName() in client/src/lib/utils.ts): gpt4 → ZHI 1 claude → ZHI 2 deepseek → ZHI 3 perplexity → ZHI 4 grok → ZHI 5

Environment secrets used: OPENAI_API_KEY ZHI 1 calls + Whisper STT ANTHROPIC_API_KEY ZHI 2 calls (DeepSeek key) ZHI 3 calls PERPLEXITY_API_KEY ZHI 4 calls XAI_API_KEY ZHI 5 calls GPTZERO_API_KEY AI detection ELEVEN_LABS_API_KEY Text-to-speech

Rate limiting: server/utils/RateLimiter.ts — per-model token bucket to prevent API quota exhaustion across concurrent requests.

================================================================================ PART 8: KNOWN COMPLEXITY AREAS
ChunkedRewriter.tsx (2941 lines) — The largest single file. Contains: chunk display, per-chunk rewrite, selection logic, re-rewrite dialog, custom text selection rewrite, generate-new-chunk dialog, save/download/email/print controls, and math view toggle. Split candidates: chunk display, action dialogs, output controls.

routes.ts (6245 lines) — All endpoints in one file. Most complex areas: rewrite-document (lines ~1089–1246), profile endpoints (lines ~3279–5836), and document chunking (lines ~5896–6084).

Math rendering pipeline — LaTeX delimiters ($$...$$, [...], (...)) must survive the entire round-trip: LLM output → response JSON → React render → KaTeX. mathDelimiterFixer.ts and dollarSignHandler.ts guard different failure modes. Any new LLM response pathway must pass through the sanitizer.

Humanizer rewrite job flow — Jobs are created in rewrite_jobs table → chunks processed sequentially → GPTZero polled before and after. If a chunk fails, the job status is set to 'error' but already-processed chunks are preserved. Partial results are recoverable.

MemStorage vs PostgreSQL — Server currently boots with MemStorage by default (see server/storage.ts). The PostgreSQL DatabaseStorage class exists but MemStorage is the active implementation.
